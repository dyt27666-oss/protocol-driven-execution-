# 6 阶段循环详解

> 每个 step 必经的 6 个阶段，缺一不可。

## 流程图

```
┌─────────────────────────────────────────────────────┐
│  阶段 1: 实施                                       │
│  - 写代码 / 改 schema / 写文档                      │
│  - TaskUpdate status=in_progress                    │
└──────────────────┬──────────────────────────────────┘
                   ↓
┌─────────────────────────────────────────────────────┐
│  阶段 2: self-test                                  │
│  - 跑模块级测试                                     │
│  - 必须 PASS                                        │
│  ✗ FAIL → 退回阶段 1                                │
└──────────────────┬──────────────────────────────────┘
                   ↓ PASS
┌─────────────────────────────────────────────────────┐
│  阶段 3: 独立 review                                │
│  - 派 Codex / subagent / Task                       │
│  - 不论结论跑完                                     │
│  - 输出 issue 列表                                  │
└──────────────────┬──────────────────────────────────┘
                   ↓
┌─────────────────────────────────────────────────────┐
│  阶段 4: 协商                                       │
│  - 分类: P0 必修 / P1 排版 / P2 记录 / 无效          │
│  - P0 直接修                                        │
│  - P1 批量 chips 给用户                             │
│  - P2 写 issue log                                  │
│  - 无效给反驳理由                                   │
└──────────────────┬──────────────────────────────────┘
                   ↓
┌─────────────────────────────────────────────────────┐
│  阶段 5: 修复                                       │
│  - 按协商结论改                                     │
│  - 再跑 self-test 必须 PASS                          │
│  - 如修复 ≥ 5 处 P0 → 再跑一轮 review               │
└──────────────────┬──────────────────────────────────┘
                   ↓
┌─────────────────────────────────────────────────────┐
│  阶段 6: commit                                     │
│  - commit message 标 review 来源                    │
│  - push 约定 remote                                 │
│  - TaskUpdate status=completed                      │
└─────────────────────────────────────────────────────┘
```

## 每阶段输出标准

### 阶段 1：实施

- 文件级 deliverable 清单
- 不修改 deliverable 范围（如需要修改先回阶段 0 改 step 定义）

### 阶段 2：self-test

- 必须有覆盖关键路径的 self-test 函数 / 测试用例
- 输出 PASS 信号（如 `print("self-test PASS")` 或 `pytest -q` 0 exit）
- happy path + 至少 1 个反例

### 阶段 3：独立 review

- review 报告（issue 列表）
- 每条 issue 含：描述、位置、严重程度建议、修复方向
- 不在阶段 3 直接采纳——只收集

### 阶段 4：协商

```
P0（必修）：
  - issue-1: <描述>
  - issue-2: <描述>
P1（用户排版）→ 批量 chips
  - issue-3: <描述> → 选项 A / B / C
  - issue-4: <描述> → 选项 A / B
P2（记录）：
  - issue-5: <描述> → 写入 issue log
无效：
  - issue-6: <描述> → 反驳理由：<verify 过 X 一致>
```

### 阶段 5：修复

- 按协商结论改
- self-test 再次 PASS
- 修复触及核心逻辑（≥ 5 处 P0）→ 阶段 3 再跑一轮

### 阶段 6：commit

commit message 标准模板：

```
<type>(<scope>): <subject>

- P0 修复 a 处: <要点>
- P1 按用户排版方案修复 b 处: <方案 ID>
- P2 / 无效记录在 issue log

Review: <reviewer 来源>
```

## 反例：不要这样做

- ❌ 跳过 self-test 直接 review
- ❌ review 全部采纳（包括明显误判）
- ❌ P0 不修就 commit
- ❌ commit message 只写 "fix issues"
- ❌ 修复后不跑 self-test 直接 commit
